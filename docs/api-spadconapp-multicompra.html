1) ¿Qué es MultiCompra y por qué te llega esto?

MultiCompra es una PWA (aplicación web instalable) que estamos haciendo para Diego. Sirve para que la cadena de locales (18 sucursales según ya vimos en el SP) maneje el flujo de:

  1. Las cajeras de cada local arman Órdenes de Pedido (OP) de lo que necesitan reponer.
  2. El supervisor en el depósito central las consolida en Órdenes de Compra (OC) a proveedores.
  3. Cuando llega la mercadería, se registra como Compra.

Las OP, OC y Compras se guardan en una base nueva (Supabase, aparte). Pero todo lo demás — usuarios, locales, productos, stock, costos — ya existe en el sistema de Diego (base ADCON2 en SQL Server).

En criollo
La PWA necesita preguntarle a tu SP cosas como "¿este usuario y clave son válidos?", "¿qué productos hay?", "¿cuánto stock tiene este producto en MULTIPLAZA?". Vos no tenés que escribir un SP nuevo: solo agregar ramas If @Reporte = '...' al SpAdconApp que ya existe.

Este documento es la lista de esas ramas, qué le va a mandar la PWA en cada caso y qué necesitamos que devuelva.

2) Cómo se conecta la PWA con el SP

La PWA habla con tu agente APISQL exactamente como dice tu manual:

POST https://nodoapi.ddns.net/sql
Authorization: Bearer e739a795...
Content-Type: application/json

{
  "sp": "Exec SpAdconApp @Reporte='MULTICOMPRA_AUTH', @Usuario='shirley', @Clave='1234'"
}

El agente devuelve un array JSON con el resultado del último SELECT del SP. Por eso, dentro de cada rama nueva que agregues, debe haber un único SELECT al final con las columnas que la PWA espera.

Ojo con los errores controlados
Para casos como "credenciales inválidas", "producto no encontrado", "usuario inactivo", etc., NO uses RAISERROR ni THROW. Devolvé una fila normal con una columna ok = 0 y otra mensaje = '...'. Así el frontend lo maneja como respuesta normal sin tener que parsear errores de SQL Server.

3) Convención de nombres de reportes

Para no chocar con tus reportes existentes (VENTASXDIA, KONTAR_PRODUCTOS, etc.) y para que sea fácil identificar cuáles son nuestros, todos los reportes nuevos empiezan con el prefijo MULTICOMPRA_:

Reporte Estado Para qué sirve
MULTICOMPRA_AUTH Especificado Validar usuario + clave al hacer login
MULTICOMPRA_LOCALES ✓ Confirmado por Carlos Lista de depósitos (un depósito = un "local" en MultiCompra)
MULTICOMPRA_PROVEEDORES Especificado Lista de proveedores activos
MULTICOMPRA_PRODUCTOS ✓ Confirmado por Carlos Catálogo completo + último proveedor por producto (para armar OC por proveedor)
MULTICOMPRA_RECALCULO_STOCK ✓ Confirmado por Carlos Recalcula la tabla snapshot de existencias (proceso pesado, se ejecuta cada X minutos)
MULTICOMPRA_STOCK ✓ Confirmado por Carlos Lee la matriz completa: IDDeposito, IDProducto, Existencia

4) MULTICOMPRA_AUTH — Login de usuarios

@Reporte = 'MULTICOMPRA_AUTH'

¿Para qué sirve?

Cuando una cajera, supervisora o admin abre la PWA, lo primero que ve es una pantalla de login con campo "Usuario" y "Clave". Cuando le da Enter, la PWA llama a este reporte. Si el usuario y clave son válidos, la PWA guarda los datos del usuario (nombre, rol, local asignado) y le muestra el menú correspondiente a su rol.

¿Qué le manda la PWA?

ParámetroTipo SQLDescripción
@Reportevarchar(50)Siempre 'MULTICOMPRA_AUTH'
@Usuariovarchar(50)Login que escribió la persona
@Clavevarchar(100)Clave en texto plano (la PWA la manda por HTTPS)

¿Qué tiene que devolver?

Una sola fila con estas columnas (los nombres tienen que ser exactos, el orden no importa):

ColumnaTipoEjemplo (éxito)Notas
okbit11 si login válido, 0 si no
idint12PK del usuario en tu tabla
nombrevarcharShirley GonzálezNombre completo para mostrar
rolvarchar(20)supervisorSolo 3 valores: cajera, supervisor, admin
mensajevarchar(100)(vacío)Mensaje de error si ok=0

Ejemplos de respuesta

Caso éxito:

[{
  "ok": 1, "id": 12, "nombre": "Shirley González", "rol": "supervisor", "mensaje": ""
}]

Caso clave incorrecta:

[{
  "ok": 0, "id": 0, "nombre": "", "rol": "", "mensaje": "Usuario o clave incorrectos"
}]

Caso usuario inactivo:

[{
  "ok": 0, "id": 0, "nombre": "", "rol": "", "mensaje": "Usuario inactivo"
}]
⚡ Actualización 28-abr-2026
Se quitaron los campos idLocal y nombreLocal del response: la tabla de usuarios del ERP de Diego no guarda asignación de usuario a local/depósito. La PWA tendrá que manejar de otra forma a qué local pertenece la sesión (ver sección 11).

Boceto T-SQL (adaptalo al schema real)

If @Reporte = 'MULTICOMPRA_AUTH' Begin

    Declare @vId      int           = 0
    Declare @vNombre  varchar(100)  = ''
    Declare @vRol     varchar(20)   = ''
    Declare @vOk      bit           = 0
    Declare @vMensaje varchar(100)  = ''
    Declare @vActivo  bit           = 0

    Select Top 1
        @vId     = U.IDUsuario,
        @vNombre = U.Nombre,
        @vRol    = U.Rol,
        @vActivo = U.Activo
    From Usuario U
    Where U.Usuario = @Usuario
      And U.Clave   = @Clave

    If @vId = 0 Begin
        Set @vMensaje = 'Usuario o clave incorrectos'
    End
    Else If @vActivo = 0 Begin
        Set @vMensaje = 'Usuario inactivo'
        Set @vId = 0
    End
    Else Begin
        Set @vOk = 1
        -- Mapear el rol original a uno de los 3 valores que espera MultiCompra
        Set @vRol = Case Upper(@vRol)
            When 'CAJERA'        Then 'cajera'
            When 'CAJERO'        Then 'cajera'
            When 'VENDEDOR'      Then 'cajera'
            When 'VENDEDOR 1'    Then 'cajera'
            When 'SUPERVISOR'    Then 'supervisor'
            When 'SUPERVISORA'   Then 'supervisor'
            When 'GERENTE'       Then 'supervisor'
            When 'ADMIN'         Then 'admin'
            When 'ADMINISTRADOR' Then 'admin'
            Else Lower(@vRol)
        End
    End

    Select
        'ok'      = Cast(@vOk as bit),
        'id'      = @vId,
        'nombre'  = @vNombre,
        'rol'     = @vRol,
        'mensaje' = @vMensaje

End
Preguntas para vos sobre AUTH
  1. ¿El login va contra la tabla Usuario o Vendedor? Las cajeras y supervisoras, ¿usan el mismo login que ya usan en el ERP?
  2. ¿La clave está en texto plano o hasheada? Si es hash, ¿qué algoritmo (SHA256, MD5)?
  3. (actualizado) Listado completo de valores reales de la columna del rol — ya vimos "VENDEDOR 1", pero pueden haber otros (CAJERA, SUPERVISOR, ADMIN, GERENTE, etc.). Para completar el CASE de mapeo a cajera / supervisor / admin.
  4. ~~¿La tabla de usuarios tiene IDDeposito?~~ Resuelto: Carlos confirmó que no existe la asignación usuario→local en el ERP. Los campos idLocal/nombreLocal se quitaron del response.

5) MULTICOMPRA_LOCALES — Lista de locales (depósitos)

@Reporte = 'MULTICOMPRA_LOCALES'
⚡ Actualizado por Carlos (28-abr-2026)
En el modelo del ERP, una Sucursal puede tener uno o varios Depósitos. La unidad operativa de MultiCompra es el Depósito, no la Sucursal. O sea: cada "local" en la PWA = un Depósito en SQL Server. La Sucursal queda como agrupador/contexto.

¿Para qué sirve?

La PWA necesita saber cuáles son los locales (depósitos) activos para poblar el dropdown de "destino" cuando se hace una OP, mostrar a qué locales se distribuye una OC, calcular stock por local, etc. Se llama una vez al iniciar sesión y se cachea.

¿Qué le manda la PWA?

ParámetroTipo SQLDescripción
@Reportevarchar(50)Siempre 'MULTICOMPRA_LOCALES'

No hace falta ningún otro parámetro.

¿Qué tiene que devolver?

Una fila por cada depósito, con estas columnas:

ColumnaTipoEjemploNotas
idint1Deposito.IDDeposito
nombrevarcharDEP MULTIPLAZADeposito.Descripcion
idSucursalint1Deposito.IDSucursal
SucursalvarcharMULTIPLAZASucursal.Descripcion (mediante JOIN)
activobit1Deposito.Estado

Ejemplo de respuesta

[
  {"id": 1, "nombre": "DEP MULTIPLAZA", "idSucursal": 1, "Sucursal": "MULTIPLAZA", "activo": 1},
  {"id": 2, "nombre": "DEP KIWI SHOP",  "idSucursal": 2, "Sucursal": "KIWI SHOP",  "activo": 1},
  {"id": 3, "nombre": "DEP GONDOLA",    "idSucursal": 3, "Sucursal": "GONDOLA",    "activo": 1},
  ...
]

T-SQL definitivo (lo pasó Carlos)

If @Reporte = 'MULTICOMPRA_LOCALES' Begin

    Select
        'id'         = D.IDDeposito,
        'nombre'     = D.Descripcion,
        'idSucursal' = D.IDSucursal,
        'Sucursal'   = S.Descripcion,
        'activo'     = D.Estado
    From Deposito D
    Join Sucursal S On D.IDSucursal = S.IDSucursal

End
Implicancias para el resto del documento
  • STOCK_PRODUCTO tiene que pivotar por IDDeposito, no por IDSucursal. Lo actualicé en su sección.
  • AUTH: el usuario tendría que tener IDDeposito asignado (no IDSucursal) para saber a qué local pertenece. Confirmar con Carlos.
  • El frontend va a filtrar por activo = 1 del lado JS, así también sabemos cuáles existen aunque estén deshabilitados.

7) MULTICOMPRA_PRODUCTOS — Catálogo de productos

@Reporte = 'MULTICOMPRA_PRODUCTOS'

¿Para qué sirve?

Es la lista maestra de productos que la PWA usa para autocompletar cuando una cajera escanea un código o escribe parte de la descripción. Esto es el catálogo "puro" (sin stock); el stock va en otro reporte aparte.

Es el catálogo maestro que la PWA usa para autocompletar cuando una cajera escanea un código o escribe parte de la descripción. Incluye también el último proveedor que vendió cada producto — eso permite al supervisor armar las OC agrupadas por proveedor automáticamente.

⚡ Actualizado por Carlos (28-abr-2026)
Carlos pasó la rama definitiva con un detalle muy útil: usa OUTER APPLY para traer el último proveedor de cada producto (mirando la última Compra registrada). Con eso la PWA no necesita un reporte aparte para "¿de quién compro habitualmente esto?" — viene en el mismo catálogo.

T-SQL definitivo (lo pasó Carlos)

If @Reporte = 'MULTICOMPRA_PRODUCTOS' Begin

    Select
        'id'          = I.IDItem,
        'codigo'      = I.CodigoBarra,
        'descripcion' = I.Descripcion,
        'categoria'   = IsNull(C.Descripcion, ''),
        'marca'       = '',
        'costo'       = IsNull(I.Costo, 0),
        'precio'      = IsNull(I.Precio, 0),
        'iva'         = IsNull(I.IVA, '10'),
        'activo'      = I.Estado,
        'idProveedor' = IsNull(UltimoProveedor.IDProveedor, '')
    From Item I
    Left Outer Join Clasificacion C On I.IDClasificacion = C.IDClasificacion
    Left Outer Join Marca M         On I.IDMarca         = M.IDMarca
    Outer Apply (
        Select Top 1
            'IDProveedor' = P.IDProveedor
        From Detalle D
        Join Compra CO    On D.IDTransaccion = CO.IDTransaccion
        Join Proveedor P  On CO.IDProveedor   = P.IDProveedor
        Where D.IDItem = I.IDItem
        Order By CO.FechaCompra Desc, CO.IDTransaccion Desc
    ) UltimoProveedor

End
Por qué viene idProveedor en el catálogo
Cuando el supervisor convierte varias OPs en OCs, MultiCompra puede agrupar automáticamente los productos por idProveedor y crear una OC por proveedor en lugar de una manual por cada uno. Sin este campo habría que hacer una llamada extra al backend por cada producto.

8) MULTICOMPRA_RECALCULO_STOCK — Recalcular snapshot de existencias

@Reporte = 'MULTICOMPRA_RECALCULO_STOCK'
⚡ Diseño propuesto por Carlos (28-abr-2026)
En vez de calcular existencia "al vuelo" cada vez que la PWA pide stock (lento y caro), se mantiene una tabla snapshot de existencia consolidada. Esta operación es la que actualiza ese snapshot. Después la PWA llama a MULTICOMPRA_STOCK que solo lee la tabla actualizada (rápido).

¿Para qué sirve?

Recalcula la existencia consolidada (sumando movimientos de Compra, Venta, Transferencia, Ajuste, etc.) y deja la tabla snapshot actualizada al momento. Lo va a llamar la PWA:

  • Cuando un usuario inicia sesión (para asegurar que ve datos frescos).
  • Después de registrar una compra (cuando implementemos MULTICOMPRA_COMPRA_REGISTRAR).
  • Periódicamente cada X minutos (configurable en frontend, ej. 15 min) si la sesión está abierta mucho tiempo.

¿Qué le manda la PWA?

ParámetroTipo SQLDescripción
@Reportevarchar(50)Siempre 'MULTICOMPRA_RECALCULO_STOCK'

¿Qué tiene que devolver?

Una sola fila confirmando que el recálculo terminó (para que el frontend sepa que ya puede llamar a MULTICOMPRA_STOCK):

ColumnaTipoEjemploNotas
okbit11 si terminó OK, 0 si falló
mensajevarchar(200)Recálculo OKTexto descriptivo (puede incluir cantidad de filas, tiempo, etc.)
fechaRecalculodatetime2026-04-28T18:05:00Timestamp del momento en que terminó

Ejemplo de respuesta

[{
  "ok": 1,
  "mensaje": "Recálculo OK — 8421 productos × 18 depósitos en 2.3 segundos",
  "fechaRecalculo": "2026-04-28T18:05:00"
}]
Notas para Carlos
  • La rama T-SQL la armás vos como te resulte más cómodo. Probablemente sea un TRUNCATE TABLE StockSnapshot seguido de un INSERT INTO ... SELECT con la suma de movimientos. O un MERGE.
  • Si el proceso es muy pesado para correrlo en cada login, podés agregar lógica tipo "si el último recálculo fue hace menos de 5 minutos, no hagas nada y devolvé OK con el timestamp anterior".
  • Si alguna vez falla a mitad, devolvé ok = 0 y el mensaje del error en mensaje — sin levantar excepción de SQL.

9) MULTICOMPRA_STOCK — Matriz completa de existencias

@Reporte = 'MULTICOMPRA_STOCK'

¿Para qué sirve?

Devuelve la matriz completa de existencias: una fila por cada combinación depósito × producto donde haya stock distinto de cero. Con esto la PWA puede hacer cualquier "cruce" en memoria sin volver a llamar al backend (ej: mostrar grilla de productos por local, calcular faltantes, sugerir transferencias, etc.).

Se llama después de MULTICOMPRA_RECALCULO_STOCK para tener datos frescos.

¿Qué le manda la PWA?

ParámetroTipo SQLDescripción
@Reportevarchar(50)Siempre 'MULTICOMPRA_STOCK'

¿Qué tiene que devolver?

Las 3 columnas que pidió Carlos:

ColumnaTipoEjemploNotas
IDDepositoint1FK a Deposito (el "local" en MultiCompra)
IDProductoint1FK a Item (mismo IDItem que devuelve MULTICOMPRA_PRODUCTOS)
Existenciadecimal8Cantidad disponible
¿Devolver solo > 0 o todas las combinaciones?
Recomendación: devolver solo filas con Existencia > 0. La PWA asume que todo lo que no aparece en la matriz está en 0. Esto reduce dramáticamente el tamaño del response (de ~180 mil filas a quizás 20-30 mil) y la PWA igual puede hacer todos los cruces porque conoce la lista completa de depósitos (de MULTICOMPRA_LOCALES) y de productos (de MULTICOMPRA_PRODUCTOS).

Ejemplo de respuesta

[
  {"IDDeposito": 1, "IDProducto": 1,    "Existencia": 8},
  {"IDDeposito": 1, "IDProducto": 7,    "Existencia": 23},
  {"IDDeposito": 1, "IDProducto": 145,  "Existencia": 2},
  {"IDDeposito": 2, "IDProducto": 1,    "Existencia": 5},
  {"IDDeposito": 2, "IDProducto": 12,   "Existencia": 18},
  ...
]

Boceto T-SQL

If @Reporte = 'MULTICOMPRA_STOCK' Begin

    Select
        'IDDeposito' = IDDeposito,
        'IDProducto' = IDItem,
        'Existencia' = Existencia
    From StockSnapshot         -- nombre real de la tabla snapshot
    Where Existencia > 0       -- omitir filas en 0 para reducir tamaño
    Order By IDDeposito, IDItem

End
Preguntas sobre STOCK
  1. ¿Cómo se llama la tabla snapshot que va a actualizar MULTICOMPRA_RECALCULO_STOCK? ¿Existe ya o la creás nueva?
  2. ¿Filtramos por Existencia > 0 o queremos también las filas en 0? (Mi recomendación: filtrar.)
  3. ¿El concepto es "stock físico" o "stock disponible" (físico - reservas)? Si hay diferencia, ¿cuál preferís devolver?

  2. Yo: mientras tanto armo una pantalla de prueba (test-api.html) que llama al endpoint con cada operación y muestra el resultado. Así apenas implementes una rama, validamos en segundos.
  3. Validamos juntos por WhatsApp/llamada, una operación a la vez. La más bloqueante es AUTH (sin login no podemos probar nada más).
  4. Cuando las 6 operaciones de lectura andan, te paso la spec de la última: MULTICOMPRA_COMPRA_REGISTRAR (cuando se confirma una compra real desde MultiCompra, registrar el ingreso de stock al sistema y disparar un recálculo).
